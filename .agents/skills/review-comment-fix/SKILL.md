## 処理内容

- PRレビューのコメントを概要として簡潔にまとめる。
- PRレビューコメントを満たすための設計案を作成する。修正が不要な場合は、その旨を記載する。  
- 設計案に基づき、変更対象ファイルを整理する。
  設計書(`docs`)についても修正要否を判断し、修正対象及び、修正内容を明記する。
- 設計案に基づき、テスト計画を整理する。
  実装するテスト項目及び、テスト方法を決定する。
  Go言語を修正した場合は、 `go test ./...` を実行する
- 設計案に基づき、リスクを整理する。
  リスクには、設計案の不備や、テスト計画の不備、実装上の不具合などが含まれる。
- 設計後、セルフレビューを行い、設計内容に不備がないか確認し、必要に応じて修正する。
- 設計内容をもとに、ソースコードの修正を行う
- 設計内容に基づき、テストを実施する。 
  テストで失敗した場合は、テスト結果をもとに修正を行う。
  テスト及び修正は最大3回まで行い、3回目で失敗した場合は残課題としてまとめる。
- `必須出力形式` に従って、実装内容をまとめる。

## 必須出力形式
次の見出しで結果を出力する。

```
## 概要
## 変更内容
## テスト結果
### `テストコマンド`
**ここに結果を表示する**
## 残課題
```
